# Instrumenting the RAG Pipeline with OpenTelemetry

Before importing `starter`, I configured an OpenTelemetry `TracerProvider` with a `ConsoleSpanExporter` so that every span generated by the application could be captured from the beginning.

```python
provider = TracerProvider()
provider.add_span_processor(SimpleSpanProcessor(console_exporter))
trace.set_tracer_provider(provider)

tracer = trace.get_tracer("llm-zoomcamp")
```

I then created a class `RAGTraced` that extends `RAGBase`. The methods `rag()`, `search()`, and `llm()` are wrapped inside their own tracing spans using `tracer.start_as_current_span(...)`.

One of the nice things about OpenTelemetry is that parent-child relationships are created automatically. Since `search()` and `llm()` are executed inside the `rag()` span, they automatically become child spans without any additional code.

Besides measuring execution time, I also stored useful metadata such as token usage and API cost by attaching them as span attributes with `span.set_attribute(...)`.

To persist traces, I implemented a custom `SQLiteSpanExporter`. Whenever a span finishes, it writes the span name, timestamps, and attributes into a `spans` table inside `traces.db`. Both the console exporter and SQLite exporter run together through separate span processors.

# Q1. First Trace — How Many Spans Are Created?

**Answer:** **3**

## Explanation

The RAG workflow consists of three traced methods:

- `rag()`
- `search()`
- `llm()`

Since each method creates exactly one span, the trace contains a total of three spans.

## Code

```python
num_spans = len(console_exporter.spans)

print("Q1 - Number of spans:", num_spans)
```

## Key Takeaway

Finished spans are exported in the order they complete rather than the order they start.

That means the exported sequence is:

```text
search
llm
rag
```

instead of

```text
rag
search
llm
```

The ordering changes, but the total number of spans remains the same.

# Q2. Capturing Metrics as Span Attributes — How Many Input Tokens?

**Answer:** **7,111 input tokens** (closest homework option: **7,000**)

## Explanation

Inside the `llm()` method, the OpenAI response includes token usage information.

I stored the input token count as a span attribute:

```python
span.set_attribute("input_tokens", response.usage.input_tokens)
```

Span attributes are a convenient way to associate additional information—such as token counts, costs, or model names—with a trace.

## Code

```python
llm_span = next(
    s for s in console_exporter.spans
    if s.name == "llm"
)

input_tokens = dict(llm_span.attributes)["input_tokens"]

print("Q2 - Input tokens:", input_tokens)
```

## Key Takeaway

The prompt includes five complete lesson pages rather than small snippets, which is why the input token count reaches several thousand tokens.

# Q3. Measuring Span Duration — How Long Does the LLM Call Take?

**Answer:** **Over 2000 ms**

Measured values from my run:

- `llm` → **9,619.7 ms**
- `search` → **126.5 ms**

## Explanation

Each span records its start and end timestamps in nanoseconds.

The duration is calculated as:

```text
(end_time - start_time) / 1_000_000
```

to convert nanoseconds into milliseconds.

## Code

```python
for span in console_exporter.spans:
    duration_ms = (
        span.end_time - span.start_time
    ) / 1_000_000

    print(
        f"{span.name}: {duration_ms:.1f} ms"
    )
```

## Key Takeaway

The LLM call is by far the slowest step in the pipeline.

While searching the local index completes in around a hundred milliseconds, the LLM call requires a network request and processes thousands of input tokens, making it several orders of magnitude slower.

Although latency varies between runs, it consistently stays well above the 2-second threshold required by the homework.

# Q4. Saving Traces into SQLite — Which Span Names Are Stored?

**Answer:**

- `rag`
- `search`
- `llm`

## Explanation

The custom `SQLiteSpanExporter` inserts one row into the database for every completed span.

As a result, every span generated during execution appears in the `spans` table.

## Code

```python
df = pd.read_sql(
    "SELECT * FROM spans",
    sqlite3.connect("traces.db")
)

span_names = sorted(
    df["name"].unique().tolist()
)

print(span_names)
```

# Q5. Querying Trace Data — Which Non-RAG Span Takes the Most Time?

**Answer:** **llm**

Measured totals:

- `llm` → **2,604.5 ms**
- `search` → **35.1 ms**

## Explanation

The `rag` span already includes the execution time of both child spans, so it should be excluded when comparing individual operations.

Grouping the remaining spans by name clearly shows that the LLM step dominates the overall execution time.

## Code

```python
df["duration_ms"] = (
    df["end_time"] - df["start_time"]
) / 1_000_000

totals = (
    df[df["name"] != "rag"]
    .groupby("name")["duration_ms"]
    .sum()
)

print(totals)
```
